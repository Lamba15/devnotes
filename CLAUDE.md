# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Core Rule

Docs under `docs/` are the source of truth for product intent; code is the source of truth for implementation. Do not silently invent product behavior when intent is unclear — read the relevant doc first, and if a rule is missing, clarify or document it before baking it into code. See `AGENTS.md` and `docs/README.md` for the full index.

## Stack

Laravel 13 + Inertia + React 19 + TypeScript + Vite 8 + Tailwind v4. PHP 8.3, Pest 4 for tests, Laravel Dusk for browser tests, Pint for PHP formatting, ESLint + Prettier for JS/TS. Laravel Wayfinder generates typed TS route/action helpers (see `resources/js/wayfinder`, `resources/js/routes`, `resources/js/actions`).

## Common commands

Dev server (runs `php artisan serve`, queue, pail logs, and vite concurrently):

```
composer dev
```

Frontend only / build / lint / types:

```
npm run dev
npm run build
npm run lint          # eslint --fix
npm run lint:check
npm run format        # prettier resources/
npm run types:check   # regenerates wayfinder then tsc --noEmit
```

PHP formatting and full CI gate:

```
composer lint         # pint --parallel
composer lint:check
composer ci:check     # lint:check + format:check + types:check + tests
```

Tests (Pest):

```
composer test                          # pint check + full pest suite
./vendor/bin/pest                      # run all
./vendor/bin/pest tests/Feature/X.php  # single file
./vendor/bin/pest --filter="name"      # single test by name
```

Browser tests (Dusk) — must target the dedicated Dusk DB, never the primary `devnotes` DB:

```
composer test:browser   # wraps scripts/test-browser.sh
```

## TDD expectation

The project follows strict TDD: write a failing Pest test first, implement the minimum to pass, refactor, rerun. Use Pest for normal application tests and Dusk for browser flows. Dusk is hard-configured to fail fast if the browser env points at the primary app database.

## Architecture

**Domain shape** (see `docs/domain/core-model.md` and `docs/product/overview.md`): `devnotes` is a personal professional operating system. The top-level entities are **Organization → Client → Project → (Issues, Boards, Transactions, Invoices, Secrets)**. There is no team model. Issues exist independently of boards; a board is a view over a project's issues and a single issue can appear on many boards. A board's backlog is computed as "issues not placed on this board." Board columns may or may not update issue status.

**Permissions** (see `docs/permissions/scopes.md`): scoped and explicit. Platform owners, client memberships (with explicit `ClientMembershipPermission`s), and project memberships are distinct. Scope and capability checks are meant to be centralized so routes, controllers, UI, and AI all go through the same checks. **AI runs with the active user's permissions — no hidden elevation.**

**Code layout:**

- `app/Models` — Eloquent models. Domain groups: tracking (`Issue`, `Board`, `BoardColumn`, `BoardIssuePlacement`, `IssueComment`), client/project (`Client`, `Project`, `ProjectMembership`, `ClientMembership*`, `SecretEntry`), finance (`Invoice`, `InvoiceItem`, `InvoiceDiscount`, `Transaction`), platform (`User`, `AuditLog`, `Attachment`, `Behavior`), and AI (`Assistant*`).
- `app/Actions` — single-purpose application actions (the preferred place for domain mutations).
- `app/Http/Controllers` — thin Inertia controllers.
- `app/Policies` — authorization; keep capability logic here and in `app/Support` rather than inline in controllers.
- `app/AI` — the central assistant. `AssistantToolRegistry` groups tools into skills; `AssistantToolExecutor` runs them; `AssistantConversationService` drives threads; `OpenRouterAssistantModelClient` / `ConfiguredFakeAssistantModelClient` implement `Contracts/AssistantModelClient`. **Every meaningful system capability should be exposed as an assistant tool**, and sensitive tools should require confirmation via `AssistantConfirmationService`.
- `resources/js/pages` — Inertia pages, one folder per top-level area (`clients`, `projects`, `boards`, `issues`, `tracking`, `finance`, `audit-logs`, `settings`, `auth`, `overview.tsx`).
- `resources/js/{components,layouts,hooks,lib}` — shared UI. Radix primitives + shadcn-style components, dnd-kit for board DnD, TipTap for rich text, Lucide for icons (used consistently across the app).
- `resources/js/{routes,actions,wayfinder}` — generated by `php artisan wayfinder:generate --with-form`; re-run via `npm run types:check` when routes or form requests change.
- `routes/web.php`, `routes/settings.php`, `routes/console.php` — HTTP + settings + console route files.

**Dates & times** (see `docs/domain/dates-and-times.md`): instants are stored as UTC timestamps and rendered in the active user's timezone. Date-only fields stay as `YYYY-MM-DD` with no timezone conversion and use the shared non-native date picker in forms.

**Finance / PDFs:** invoice and transaction PDFs use `barryvdh/laravel-dompdf`. Invoice statuses are `draft | pending | paid | overdue`; both invoices and transactions carry a currency field.

**AI credit system:** non-owner access is gated by credits — `-1` unlimited, `0` none, positive integer is a quota.

## Conventions worth knowing

- Prefer updating an existing doc over creating duplicates; keep stable docs in `docs/` separate from working drafts in `docs/working/`.
- Favor explicit behavior over vague role labels and framework defaults. Avoid generic SaaS defaults unless intentional.
- Keep scope/capability checks centralized so routes, UI, and AI tools reuse the same logic.
- When editing routes, form requests, or adding new Inertia actions, run `npm run types:check` to regenerate Wayfinder output before committing.
